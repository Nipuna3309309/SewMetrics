 PART 1 — SYSTEM OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WHAT THE SYSTEM DOES:
  The SewMetrics system monitors a sewing machine in real time using 4 physical
  sensors. The previous design ran 4 separate ML models, one for each sensor.
  This redesign replaces them with exactly 2 models that cover all 4 sensors
  together, each model serving a distinct and complementary purpose:

    Model 1 — Isolation Forest   : "Is the machine behaving abnormally RIGHT NOW?"
    Model 2 — Linear Regression  : "Is the machine slowly getting WORSE over time?"

  Together, these two questions cover the full scope of predictive maintenance:
    - Short-term fault detection (Model 1)
    - Long-term degradation monitoring (Model 2)

SENSORS IN THE SYSTEM:
  ┌─────────────────┬──────────────────────────────────────────────────────┐
  │ Sensor          │ What it measures                                     │
  ├─────────────────┼──────────────────────────────────────────────────────┤
  │ Hall-effect     │ Motor RPM (rotations per minute via magnetic pulses) │
  │ IR sensor       │ Stitch count / stitches per minute (SPM)             │
  │ MPU6050 (IMU)   │ 3-axis vibration (accel X, Y, Z + smoothed stable)  │
  │ DHT11           │ Temperature (°C) and humidity (%)                    │
  └─────────────────┴──────────────────────────────────────────────────────┘

 PART 2 — WHY REPLACE 4 MODELS WITH 2 MODELS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PROBLEM WITH THE 4-MODEL APPROACH:

  1. SENSOR BLINDNESS
     Each model only sees ONE sensor at a time. In a real machine, components
     INTERACT. A failing bearing causes BOTH high vibration AND rising temperature
     AND inconsistent RPM simultaneously. A model watching only vibration will
     flag it. A model watching only RPM may not. A model watching only temperature
     may not. No single sensor tells the full story.

     Example: Thread jam scenario:
       - RPM stays normal (motor is still spinning)
       - Temperature stays normal (no extra friction yet)
       - Vibration stays normal (no mechanical imbalance)
       - Only SPM drops to zero
     The RPM model says "normal". The temperature model says "normal".
     The vibration model says "normal". The stitch model ALONE catches it.
     But if we had ONE model looking at ALL features together, it immediately
     sees the contradiction: RPM=1400, SPM=0, ratio=0 — that is a massive anomaly.

  2. NO CROSS-SENSOR CORRELATION
     The most powerful predictive signals are RATIOS and RELATIONSHIPS between
     sensors, not the raw sensor values themselves. For example:
       - SPM/RPM ratio dropping → thread problem or sensor misalignment
       - Temperature rising while RPM is stable → cooling problem, not overload
       - Vibration rising while SPM drops → needle/hook mechanical issue
     None of these cross-sensor insights are possible with 4 separate models.

  3. REDUNDANT ARCHITECTURE
     4 models × 4 endpoints × 4 database queries = 4× the code, 4× the
     maintenance, 4× the potential failure points. The 2-model design does
     more with less.

  4. NO TEMPORAL AWARENESS
     The 4 models only look at current-state anomalies. None of them detect
     GRADUAL degradation — the slow, months-long wear that precedes most
     real machine failures. A machine with vibration slowly increasing from
     100 to 200 to 300 raw units over 2 weeks is NOT anomalous at any single
     reading, but IS degrading. Only a dedicated trend model catches this.

WHY 2 MODELS IS THE RIGHT NUMBER:
  - 1 model cannot serve both real-time detection AND long-term trend analysis
    simultaneously — they require different input windows and different logic
  - 2 models is the minimum complete solution:
    one for NOW (anomaly), one for OVER TIME (degradation)
  - More than 2 models re-introduces the complexity we are removing

 PART 3 — HOW ALL 4 COMPONENTS ARE STILL COVERED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The key insight is this: the 4 sensor components are not removed. They are
  CONVERTED INTO FEATURES and fed together into both models.

  Each sensor contributes engineered features (not raw values) to both models:

  ┌───────────────────┬────────────────────────────────────────────────────┐
  │ Original Model    │ Contribution in New 2-Model Design                 │
  ├───────────────────┼────────────────────────────────────────────────────┤
  │ RPM Analyzer      │ → avg_rpm, rpm_std features in Model 1 (IF)        │
  │ (EWMA anomaly)    │ → rpm_stability_trend feature in Model 2 (LR)      │
  ├───────────────────┼────────────────────────────────────────────────────┤
  │ Stitches Analyzer │ → avg_spm, spm_to_rpm_ratio features in Model 1    │
  │ (Linear forecast) │ → spm_efficiency_trend feature in Model 2          │
  ├───────────────────┼────────────────────────────────────────────────────┤
  │ Vibration Analyzer│ → vibration_rms, vibration_std features in Model 1 │
  │ (Isolation Forest)│ → vibration_level_trend feature in Model 2         │
  ├───────────────────┼────────────────────────────────────────────────────┤
  │ Temperature       │ → avg_temp, temp_rise_rate, humidity in Model 1    │
  │ Analyzer (dT/dt)  │ → temperature_trend feature in Model 2             │
  └───────────────────┴────────────────────────────────────────────────────┘

  Every sensor's information is preserved. What is removed is the isolation —
  sensors now INFORM each other rather than operating in silos.

 PART 4 — COMPLETE DATA FLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PHYSICAL LAYER (Hardware):
  ESP32 reads sensors every ~5 seconds
    → Hall-effect  → rpm_live, rpm_1min, current_minute_revs
    → IR sensor    → stitches_total, spm
    → MPU6050      → accel_x, accel_y, accel_z, vibration_raw, vibration_stable
    → DHT11        → temperature_c, humidity
  ESP32 publishes JSON to MQTT topics:
    factory/machine/SM_01/rpm
    factory/machine/SM_01/stitches
    factory/machine/SM_01/vibration
    factory/machine/SM_01/temperature

STORAGE LAYER (Node.js → MongoDB):
  Node.js backend receives MQTT messages
  Saves each message as a document in sensor_readings collection
  65,000+ documents stored, indexed by machine_id + sensor + received_at

FEATURE ENGINEERING LAYER (Python — the bridge):
  When analytics endpoint is called, fetch last N readings per sensor type:
    rpm_docs     → last 60 RPM readings
    stitch_docs  → last 60 stitch readings
    vib_docs     → last 60 vibration readings
    temp_docs    → last 60 temperature readings

  From these 4 sets of readings, compute a FEATURE WINDOW:
  For each rolling window of W readings, compute one feature vector:
    [avg_rpm, rpm_std, avg_spm, spm_to_rpm, vib_rms, vib_std,
     avg_temp, temp_rise_rate, humidity]

  For Model 1 (short window, W=15 readings ≈ last 75 seconds):
    Compute feature vector for each window → build feature MATRIX X
    X has shape (N_windows, 9 features)
    The most recent window is the "current state"

  For Model 2 (long window, W=60 readings per bucket, N=10 buckets):
    Divide historical data into 10 time buckets
    Each bucket produces one summary vector
    These 10 vectors show how health indicators CHANGE over time

MODEL 1: ISOLATION FOREST (Anomaly Detection)
  Input  : Feature matrix X  (shape: N_windows × 9 features)
  Process: Train IsolationForest on all windows, score the latest window
  Output :
    - anomaly_score   (0-1, higher = more anomalous)
    - anomaly_flag    (True if score > threshold)
    - contributions   (which features deviated most from normal)
    - component_flags (which sensor components are most responsible)

MODEL 2: LINEAR REGRESSION (Degradation Trend)
  Input  : Time series of health indicators across N buckets
  Process: Fit linear regression to each health indicator over time
           Combine slopes into a degradation rate
  Output :
    - degradation_rate (positive = getting worse, negative = improving)
    - health_trend_score (0-100, 100 = no degradation)
    - per-feature trends (vibration_trend, temp_trend, etc.)
    - early_warning_flag (True if degradation rate > threshold)
    - projected_failure_days (rough estimate based on trend slope)

DASHBOARD LAYER (React Frontend):
  System Health Page: Uses both models for full machine overview
  Sensor Pages: Show this sensor's specific feature contributions from Model 1
               + this sensor's individual trend from Model 2

 PART 5 — FEATURE SET (Complete Specification)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MODEL 1 FEATURES — 9 FEATURES (Isolation Forest input vector):

  ┌────┬──────────────────────┬────────────────────────────────────────────────────────┐
  │ #  │ Feature Name         │ Calculation & Why It Matters                           │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F1 │ avg_rpm              │ mean(rpm_live[-W:])                                     │
  │    │                      │ Primary operational speed. Abnormal if machine is       │
  │    │                      │ running too fast/slow or completely stopped.            │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F2 │ rpm_std              │ std(rpm_live[-W:])                                      │
  │    │                      │ Stability of motor speed. High std = unstable power     │
  │    │                      │ delivery or mechanical binding causing speed surges.    │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F3 │ avg_spm              │ mean(spm[-W:])                                          │
  │    │                      │ Production rate. Direct measure of machine output.      │
  │    │                      │ Drop in SPM = thread break, needle issue, operator.     │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F4 │ spm_to_rpm_ratio     │ avg_spm / (avg_rpm + ε)                                 │
  │    │                      │ THE MOST POWERFUL CROSS-SENSOR FEATURE.                 │
  │    │                      │ In a healthy machine, stitches are proportional to RPM.│
  │    │                      │ If RPM=1400 but SPM=0 → ratio=0 → massive anomaly.     │
  │    │                      │ Catches thread jams that single-sensor models miss.     │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F5 │ vibration_rms        │ sqrt(mean(vibration_stable[-W:]²))                      │
  │    │                      │ Root Mean Square vibration energy. RMS is the standard  │
  │    │                      │ industrial metric for vibration severity — used in ISO  │
  │    │                      │ 10816 machine vibration standards.                      │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F6 │ vibration_std        │ std(vibration_stable[-W:])                              │
  │    │                      │ Consistency of vibration. A machine with CONSTANT high  │
  │    │                      │ vibration may be normal for that machine. A machine     │
  │    │                      │ with ERRATIC vibration (high std) is malfunctioning.    │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F7 │ avg_temp             │ mean(temperature_c[-W:])                                │
  │    │                      │ Average thermal state. Elevated temp = friction,        │
  │    │                      │ poor ventilation, or overloading.                       │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F8 │ temp_rise_rate       │ polyfit(indices, smoothed_temps[-20:], 1)[0] × 12       │
  │    │                      │ Rate of temperature change in °C per minute.            │
  │    │                      │ A RISING temp is more dangerous than a HIGH temp.       │
  │    │                      │ Rate > 0.5°C/min is an early warning — still below      │
  │    │                      │ threshold but heading toward it fast.                   │
  ├────┼──────────────────────┼────────────────────────────────────────────────────────┤
  │ F9 │ humidity             │ latest humidity reading (%)                             │
  │    │                      │ High humidity → insulation resistance drops on          │
  │    │                      │ motor windings → abnormal current → affects RPM.        │
  │    │                      │ Humidity > 70% combined with high temp is critical.     │
  └────┴──────────────────────┴────────────────────────────────────────────────────────┘

MODEL 2 FEATURES — 5 TREND INDICATORS (Linear Regression inputs):

  These are NOT raw sensor values. Each is a STATISTIC computed over a
  time bucket (window of ~60 readings = ~5 minutes of data).
  We compute 10-20 such buckets across the available history, then
  fit a line through each indicator over time.

  ┌────┬──────────────────────────┬──────────────────────────────────────────────────────┐
  │ #  │ Trend Indicator          │ Calculation                                          │
  ├────┼──────────────────────────┼──────────────────────────────────────────────────────┤
  │ T1 │ vibration_level          │ mean(vibration_stable) per bucket                    │
  │    │                          │ Rising over time → bearing wear or imbalance growing  │
  ├────┼──────────────────────────┼──────────────────────────────────────────────────────┤
  │ T2 │ temperature_level        │ mean(temperature_c) per bucket                       │
  │    │                          │ Slowly rising baseline → cooling degradation          │
  ├────┼──────────────────────────┼──────────────────────────────────────────────────────┤
  │ T3 │ spm_efficiency           │ mean(spm) / (mean(rpm) + ε) per bucket               │
  │    │                          │ Falling ratio over time → mechanical wear in needle   │
  │    │                          │ mechanism or increasing thread drag over sessions     │
  ├────┼──────────────────────────┼──────────────────────────────────────────────────────┤
  │ T4 │ rpm_stability            │ 1 / (std(rpm) + 1) per bucket                        │
  │    │                          │ INVERSE of RPM variability. If RPM becomes more       │
  │    │                          │ erratic over time (std rising), rpm_stability falls.  │
  │    │                          │ Declining stability → power delivery degrading.       │
  ├────┼──────────────────────────┼──────────────────────────────────────────────────────┤
  │ T5 │ composite_anomaly_score  │ IF anomaly score per bucket (from Model 1 applied    │
  │    │                          │ to each historical bucket separately)                 │
  │    │                          │ RISING anomaly scores over time = machine is getting  │
  │    │                          │ progressively harder to classify as normal.           │
  └────┴──────────────────────────┴──────────────────────────────────────────────────────┘

  NOTE ON COMPOSITE ANOMALY SCORE (T5):
  This is the most powerful trend indicator. We run Model 1's feature
  engineering on each historical bucket, then score that feature vector
  using the Isolation Forest. The resulting score sequence shows whether
  the machine was more anomalous 1 hour ago vs 2 hours ago vs 4 hours ago.
  A rising sequence of anomaly scores over hours/days is the clearest signal
  of gradual degradation — even if no single reading ever crossed a threshold.

 PART 6 — MODEL 1 DETAIL: ISOLATION FOREST
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ALGORITHM:
  Isolation Forest (Liu et al., 2008). Unsupervised anomaly detection.
  No labeled failure data required. Trains in milliseconds on 60-200 samples.

CORE PRINCIPLE:
  Anomalies are rare and different, therefore they are easier to isolate.
  Build N random decision trees. For each data point, count how many
  splits are needed to isolate it (get it alone in a leaf).
  Normal points → need many splits (deep in tree)
  Anomalies    → isolated in few splits (shallow in tree)
  Anomaly score = normalized average path length across all trees.

PARAMETERS:
  n_estimators    = 100   (100 trees, enough for robust scoring)
  contamination   = 0.05  (assume 5% of readings may be anomalous)
  random_state    = 42    (reproducible results for research reports)
  max_features    = 1.0   (use all 9 features in each split)
  max_samples     = "auto" (use 256 samples per tree by default)

FEATURE IMPORTANCE (EXPLAINABILITY):
  Isolation Forest does not natively output feature importance.
  We use DEVIATION ANALYSIS as a proxy:
    1. Compute mean and std of each feature across all training windows
    2. For the latest (anomalous) feature vector, compute z-scores:
         z_i = (feature_i - mean_i) / std_i
    3. Features with |z_i| > 2 are contributing to the anomaly
    4. Rank by |z_i| to get "top contributors"
    5. Map feature names to component names for dashboard display

  Example output:
    Top contributors: [("spm_to_rpm_ratio", z=-4.2), ("vibration_rms", z=3.1)]
    → Tells operator: "The stitch-to-RPM relationship broke down and
       vibration is unusually high. Check the needle mechanism and bearings."

TRAINING STRATEGY:
  We retrain on the last 60-200 readings every call. This is called
  "rolling window retraining" or "adaptive baseline".
  Why not train once and persist?
    - Machines change behavior over shifts (warm vs cold state)
    - New baseline after maintenance should not be flagged as anomaly
    - Rolling window naturally adapts to the current machine state
  Why not train on ALL historical data?
    - A machine from 3 months ago may have been poorly tuned
    - We want anomaly = unusual for THIS machine NOW, not historically

INPUT WINDOW SIZE CHOICE (W=15 readings):
  15 readings × ~5 seconds = ~75 seconds of data per feature vector.
  This gives a "snapshot" of current machine behavior that is:
    - Long enough to compute meaningful statistics (std, mean)
    - Short enough to respond quickly to sudden changes
    - Short enough to have many training windows from 60 readings total

ANOMALY SCORE THRESHOLD:
  Default IF threshold: predict == -1 (trained contamination threshold)
  We additionally use normalized score > 0.5 as a soft "watch" level.
  Score > 0.7 → warning
  Score > 0.85 → critical

 PART 7 — MODEL 2 DETAIL: LINEAR REGRESSION DEGRADATION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ALGORITHM:
  Ordinary Least Squares Linear Regression (numpy.polyfit, degree=1).
  Applied independently to each of the 5 trend indicators over time.

CORE PRINCIPLE:
  Fit a straight line y = mt + b through each health indicator over time.
  The slope m is the KEY output:
    m > 0 for vibration/temperature/anomaly_score → getting worse
    m < 0 for spm_efficiency/rpm_stability → getting worse
  Magnitude of m = rate of degradation.

WHY LINEAR REGRESSION FOR DEGRADATION?
  1. Mechanical wear IS approximately linear over time (Archard's law of wear)
  2. It gives an interpretable slope that maps directly to "X units per hour"
  3. You can project forward: "if this trend continues, failure in Y days"
  4. No training data needed — the model fits whatever history is available
  5. R² coefficient tells you how "clean" the trend is (high R² = consistent)
  6. Perfectly explainable in a viva: "the slope is positive, meaning degrading"

BUCKET CONSTRUCTION:
  We fetch up to 500 readings per sensor (the maximum history available).
  We divide this into N_BUCKETS = 10 equal time segments.
  For each bucket, we compute the 5 trend indicators listed in Part 5.
  This gives a (10, 5) matrix — 10 time points, 5 health indicators.
  For each of the 5 columns, we fit linear regression over the 10 time points.

DEGRADATION RATE CALCULATION:
  For each indicator, we normalize its slope:
    vibration slope:       normalized = slope / VIBRATION_DANGER (3000 raw units)
    temperature slope:     normalized = slope / TEMP_DANGER (60°C)
    spm_efficiency slope:  normalized = -slope (negative slope = degradation)
    rpm_stability slope:   normalized = -slope (negative slope = degradation)
    anomaly_score slope:   normalized = slope (already 0-1 scale)

  Combined degradation rate = mean of all 5 normalized slopes
  Scale to percentage per hour: degradation_rate = combined * 100

HEALTH TREND SCORE (0-100):
  health_trend_score = max(0, min(100, 100 - degradation_rate * 10))
  Interpretation:
    100   → No degradation detected, all trends stable or improving
    80-99 → Minor degradation, monitor
    60-79 → Moderate degradation, plan maintenance soon
    40-59 → Significant degradation, schedule maintenance this week
    0-39  → Rapid degradation, urgent maintenance required

PROJECTED FAILURE ESTIMATE:
  For each indicator, if slope > 0 (degrading):
    current_value = latest bucket value
    danger_value  = threshold for that indicator
    time_to_danger = (danger_value - current_value) / (slope_per_bucket)
    time_in_hours  = time_to_danger × bucket_duration_hours
  Report minimum across all indicators as "estimated days to maintenance"

EARLY WARNING FLAG:
  Triggers when ANY of:
    - degradation_rate > 2.0 (rapid worsening)
    - health_trend_score < 70 AND anomaly_score_trend slope > 0
    - vibration trend slope projects failure within 7 days
    - temperature trend slope projects crossing 60°C within 24 hours
    - R² of any critical indicator > 0.8 (very clean trend = real degradation,
      not random noise)

 PART 8 — COMPLETE WORKFLOW (End-to-End)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 1 — DATA FETCH
  Fetch from MongoDB:
    rpm_docs     ← last 500 readings WHERE sensor = "rpm"
    stitch_docs  ← last 500 readings WHERE sensor = "stitches"
    vib_docs     ← last 500 readings WHERE sensor = "vibration"
    temp_docs    ← last 500 readings WHERE sensor = "temperature"

STEP 2 — TEMPORAL ALIGNMENT
  Sort all 4 arrays by received_at (oldest to newest).
  Pair readings by approximate time bucket (±10 seconds).
  This creates aligned windows where each row represents the same
  time period across all 4 sensors.

STEP 3 — FEATURE ENGINEERING (for Model 1)
  For each sliding window of W=15 readings (stride=1):
    Extract the 9 features listed in Part 5, Feature Table.
    Append to feature matrix X.
  X shape = (N_windows, 9)
  Latest row of X = current machine state feature vector.

STEP 4 — ISOLATION FOREST (Model 1 — Anomaly Detection)
  4a. Train:
      model = IsolationForest(n_estimators=100, contamination=0.05)
      model.fit(X)          ← trains on ALL windows including recent history
  4b. Score latest window:
      score = model.score_samples(X[-1:])  ← lower = more anomalous
      anomaly_flag = model.predict(X[-1:]) == -1
  4c. Normalize score to 0-1 (higher = more anomalous):
      normalized_score = 1 - (score - X_scores.min()) / (X_scores.ptp() + ε)
  4d. Compute feature contributions:
      z_scores = (X[-1] - X.mean(axis=0)) / (X.std(axis=0) + ε)
      top_contributors = argsort(|z_scores|)[::-1][:3]  ← top 3 deviating features
  4e. Map features to components:
      avg_rpm, rpm_std → RPM component
      avg_spm, spm_to_rpm → Stitches component
      vibration_rms, vibration_std → Vibration component
      avg_temp, temp_rise_rate, humidity → Temperature component

STEP 5 — BUCKET COMPUTATION (for Model 2)
  Divide all data into N_BUCKETS = 10 equal time segments.
  For each bucket i:
    Extract all readings within that time range for each sensor.
    Compute: vib_level_i, temp_level_i, spm_eff_i, rpm_stability_i
    Apply Model 1 feature engineering to bucket data and score with IF
    → anomaly_score_i
    Append [vib_level, temp_level, spm_eff, rpm_stability, anomaly_score] as row i
  Result: trend_matrix shape (10, 5)

STEP 6 — LINEAR REGRESSION (Model 2 — Degradation Trend)
  6a. For each of 5 columns in trend_matrix:
      slope, intercept = polyfit(range(10), trend_matrix[:, j], 1)
      r_squared = pearsonr(range(10), trend_matrix[:, j])[0]²
      Store slope and r_squared for each indicator.
  6b. Compute normalized degradation contribution from each slope.
  6c. Sum to get overall degradation_rate.
  6d. Compute health_trend_score.
  6e. Check early warning conditions.
  6f. Estimate projected_failure_days.

STEP 7 — OUTPUT AGGREGATION (System Health)
  Combine Model 1 + Model 2 outputs into final system health report:
    current_anomaly_status  ← from Model 1
    health_trend_score      ← from Model 2
    machine_health_score    ← weighted(anomaly-based=60%, trend-based=40%)
    early_warning_flag      ← OR of both model warnings
    top_issues              ← from Model 1 feature contributions
    degradation_indicators  ← from Model 2 per-feature trends
    maintenance_recommendation ← derived from both models combined

 PART 9 — DASHBOARD OUTPUTS (What the Frontend Displays)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

OUTPUT 1 — CURRENT ANOMALY STATUS
  Source:      Model 1 (Isolation Forest)
  Values:      NORMAL / WATCH / WARNING / CRITICAL
  Dashboard:   Color-coded status banner on every sensor page
  Used for:    Operator immediate action decisions

OUTPUT 2 — ANOMALY SCORE
  Source:      Model 1 (Isolation Forest)
  Range:       0.0 to 1.0
  Dashboard:   Gauge or progress bar on Analytics page
  Used for:    Understanding HOW anomalous the current state is

OUTPUT 3 — TOP CONTRIBUTING FEATURES
  Source:      Model 1 (deviation analysis on feature vector)
  Example:     "spm_to_rpm_ratio: z=-4.2, vibration_rms: z=3.1"
  Dashboard:   Ranked list on Analytics page + per-sensor page highlights
  Used for:    Root cause identification without domain expertise

OUTPUT 4 — HEALTH TREND SCORE
  Source:      Model 2 (Linear Regression)
  Range:       0 to 100 (100 = no degradation)
  Dashboard:   Trend line chart on Analytics page showing score over time
  Used for:    Long-term maintenance planning

OUTPUT 5 — DEGRADATION RATE
  Source:      Model 2 (Linear Regression)
  Values:      %/hour (positive = worsening, negative = improving)
  Dashboard:   Number with up/down arrow and color coding
  Used for:    Quantifying how fast the machine is degrading

OUTPUT 6 — PER-INDICATOR TRENDS
  Source:      Model 2 (5 individual slope values)
  Example:     vibration: +0.4/hr, temperature: +0.01°C/hr, spm_eff: -0.001/hr
  Dashboard:   Spark lines or arrows on each sensor card
  Used for:    Identifying WHICH aspect is degrading

OUTPUT 7 — EARLY WARNING FLAG
  Source:      Model 2 (threshold on degradation rate + R²)
  Values:      True / False
  Dashboard:   Alert banner with orange border on Analytics page
  Used for:    Proactive maintenance scheduling

OUTPUT 8 — PROJECTED FAILURE ESTIMATE
  Source:      Model 2 (linear extrapolation to danger threshold)
  Example:     "Estimated 7.2 hours to critical vibration threshold"
  Dashboard:   Countdown on Analytics page
  Used for:    Maintenance window planning

OUTPUT 9 — MACHINE HEALTH SCORE (MHS)
  Source:      Weighted combination of Model 1 + Model 2
  Formula:     MHS = (1 - anomaly_score) * 60 + health_trend_score * 0.4
               → 60% weight on current anomaly state (Model 1)
               → 40% weight on long-term trend health (Model 2)
  Range:       0 to 100
  Dashboard:   Circular gauge (SVG arc) as the hero metric
  Used for:    Single-number executive summary

 PART 10 — WHY THIS IS BETTER (Technical Justification)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. INDUSTRIAL ALIGNMENT
   The ISO 13374 standard for condition monitoring of machines recommends
   exactly this two-layer approach: anomaly detection + trend analysis.
   Our design matches industry best practice.

2. CROSS-SENSOR SYNERGY
   The spm_to_rpm_ratio feature alone is more diagnostic than any single
   sensor model. A ratio < 0.01 while RPM > 500 is an unambiguous thread fault.
   No 4-model design can compute this cross-sensor signal.

3. UNSUPERVISED — NO LABELED DATA NEEDED
   Neither Isolation Forest nor Linear Regression requires labeled "failure"
   examples. This is critical for a student/research project because sewing
   machine failure data with ground truth labels does not exist publicly.
   Both models learn from normal operating data only.

4. EXPLAINABILITY (CRITICAL FOR VIVA/REPORT)
   Every output can be explained in plain English:
   "The Isolation Forest found the current feature vector is in the top 5%
    most unusual states it has seen. The biggest contributor is the
    spm_to_rpm_ratio being 3.2 standard deviations below normal."
   "The Linear Regression shows vibration has been increasing by 8.4 raw
    units per hour for the last 6 hours. At this rate, it will reach the
    3000-unit threshold in approximately 7.3 hours."
   These sentences can go directly into a project report.

5. COMPUTATIONAL EFFICIENCY
   2 model calls vs 4 model calls.
   1 set of database queries vs 4 sets.
   IsolationForest on 200×9 matrix: <50ms.
   Linear regression on 10×5 matrix: <1ms.
   Total analytics response time: <200ms including MongoDB queries.

6. ARCHITECTURAL CLEANLINESS
   4 files reduced to 2 files.
   4 API endpoints reduced to 2 core endpoints (+ 1 combined system endpoint).
   Single feature engineering function used by BOTH models.
   Single database fetch shared by BOTH models.
   If a new sensor is added later, add its features to the 9-feature vector —
   neither model needs to change structurally.

7. SCIENTIFICALLY DEFENSIBLE
   Both algorithms are peer-reviewed, widely published, and used in industry:
   - Isolation Forest: Liu et al. (2008), cited 10,000+ times
   - Linear Regression: Gauss (1795), the foundation of modern statistics
   Both are available in scikit-learn, the most-used ML library in Python.
   Examiner cannot question the validity of the algorithms themselves.

    GET /analytics/stitches/:machineId
    GET /analytics/vibration/:machineId
    GET /analytics/temperature/:machineId

    GET /analytics/anomaly/:machineId
      → Runs Model 1 (Isolation Forest)
      → Returns: anomaly_score, anomaly_flag, status, top_contributors,
                 component_flags, feature_vector, feature_z_scores

      → Returns: machine_health_score, overall_status, maintenance_priority,

                 degradation_result (full Model 2 output),
                 top_issues, recommendation
