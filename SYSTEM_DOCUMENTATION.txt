 SECTION 1 — WHAT IS SEWMETRICS?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SewMetrics is a real-time IoT factory monitoring system for sewing machines.
Physical sensors attached to a sewing machine send live data over WiFi using
the MQTT protocol. That data is received by a backend server, stored in a
cloud database (MongoDB Atlas), and displayed on a web dashboard.

On top of the live monitoring, a machine learning analytics layer runs on the
stored data using exactly 2 unified ML models:

  Model 1 — Isolation Forest   "Is the machine behaving abnormally RIGHT NOW?"
  Model 2 — Linear Regression  "Is the machine slowly getting WORSE over time?"

Together these two models cover the full scope of predictive maintenance:
short-term fault detection (Model 1) and long-term degradation monitoring
(Model 2). Both models use all 4 sensors together, not one sensor each.

The system has 3 running services:
  1. Python Analytics Service  — port 5003  (2 ML models, FastAPI v2.0)
  2. Node.js Backend           — port 4000  (data ingestion + API + proxy)
  3. React Frontend            — port 3000  (dashboard UI)

 SECTION 2 — HOW DATA FLOWS THROUGH THE ENTIRE SYSTEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 1 — ARDUINO (HARDWARE)
  The sewing machine has an ESP32 microcontroller with 4 physical sensors:
    • Hall-effect sensor  → measures motor RPM (magnetic pulse counting)
    • IR sensor           → counts needle stitches (interrupt on needle pass)
    • MPU6050 (IMU)       → measures vibration via 3-axis accelerometer (I2C)
    • DHT11               → measures temperature and humidity (digital pin)

  The ESP32 reads all sensors every ~5 seconds, formats the data as JSON,
  and publishes it over MQTT (encrypted, port 8883) to HiveMQ Cloud broker.

  Example MQTT message published to topic  factory/machine/SM_01/rpm :
  {
    "machine_id": "SM_01",
    "sensor": "rpm",
    "rpm_live": 1423.5,
    "rpm_1min": 1401.0,
    "rpm_1min_ready": true,
    "minute_progress_sec": 42,
    "current_minute_revs": 980,
    "time_ms": 1843200,
    "wifi_connected": true
  }

  MQTT Topic structure: factory/machine/{machine_id}/{sensor_type}
  Four topics per machine: .../rpm  .../stitches  .../vibration  .../temperature

STEP 2 — NODE.JS BACKEND (server.js)
  The backend subscribes to wildcard topic  factory/machine/+/+
  so it receives ALL messages from ALL machines and ALL sensor types.

  On every received message:
    a) Parse JSON payload
    b) Save full document to MongoDB collection "sensor_readings"
    c) Update in-memory object latestByMachine[machineId] with latest values

  The in-memory object merges all 4 sensor types together so the frontend
  can get a single merged "snapshot" of all sensors at once.

STEP 3 — MONGODB ATLAS (Cloud Database)
  Every sensor reading is stored as a document in collection: sensor_readings
  Fields: machine_id, sensor, topic, received_at, rpm_live, rpm_1min,
          stitches_total, spm, accel_x, accel_y, accel_z, vibration_stable,
          temperature_c, humidity, wifi_connected, time_ms, raw_payload

  ~65,000+ documents stored for machine SM_01.
  Indexed by: machine_id + sensor + received_at

STEP 4 — REST API (Node.js Express)
  The backend exposes these endpoints to the frontend:

    GET /api/health
      → Returns system health: mqtt_connected, mongodb_ready

    GET /api/latest/:machineId
      → Returns the merged in-memory snapshot of ALL sensors for that machine
      → Used by dashboard for live values (polled every 2.5 seconds)

    GET /api/readings/:machineId?limit=N
      → Returns last N documents from MongoDB (any sensor type)

    GET /api/history/:machineId/:sensor?limit=N
      → Returns last N documents filtered by sensor type (rpm/stitches/etc)
      → Used by charts (polled every 4 seconds)

    GET /api/alerts/:machineId
      → Runs threshold checks on latest values, returns alert objects
      → Checks: temperature > 60°C, vibration > 3000, rpm>0 but spm=0

    GET /api/analytics/:type/:machineId
      → PROXY to Python analytics service on port 5003
      → Types: anomaly, degradation, system

STEP 5 — PYTHON ANALYTICS SERVICE (FastAPI, port 5003, version 2.0)
  Fetches historical sensor data from MongoDB for all 4 sensors at once.
  Runs shared feature engineering, then feeds into one or both ML models.
  Returns JSON with anomaly flags, degradation trends, health scores,
  failure projections, and maintenance recommendations.
  Frontend polls this every 20 seconds.

  DATA FETCH per analytics call:
    For /anomaly   → last 200 readings per sensor (4 fetches → shared)
    For /degradation → last 500 readings per sensor (for trend buckets)
    For /system    → last 300 readings per sensor (both models run internally)

STEP 6 — FEATURE ENGINEERING (Python — analytics/analyzers/features.py)
  This is the critical bridge between raw sensor docs and the ML models.
  Both models use the SAME feature engineering function.

  From 4 arrays of raw sensor docs, a sliding window of W=15 readings
  produces one 9-feature vector:

    [avg_rpm, rpm_std, avg_spm, spm_to_rpm_ratio,
     vibration_rms, vibration_std, avg_temp, temp_rise_rate, humidity]

  Multiple overlapping windows produce a FEATURE MATRIX X (shape: N × 9).
  Each row = one snapshot of the machine's state during a 75-second window.

STEP 7 — ML MODEL 1: ISOLATION FOREST (anomaly.py)
  Input : Feature matrix X — the most recent window is the "current state"
  Trains on all N windows in X (rolling window baseline = current machine)
  Scores the latest window to detect if current state is anomalous
  Returns: anomaly_score (0-1), status, top_contributors (z-scores), etc.

STEP 8 — ML MODEL 2: LINEAR REGRESSION (degradation.py)
  Input : Same 4 sensor arrays, but viewed as 10 time buckets over history
  For each bucket, compute 5 health indicators:
    vibration_level, temperature_level, spm_efficiency, rpm_stability,
    composite_anomaly_score (Model 1 applied to each historical bucket)
  Fit OLS regression through each indicator over 10 time points
  Slope = degradation rate (positive slope on vibration/temp = getting worse)
  Returns: health_trend_score (0-100), degradation_rate, projected_failure_hours

STEP 9 — SYSTEM HEALTH (system.py)
  Calls both models, combines into unified Machine Health Score (MHS):
    MHS = 50% × (100 - anomaly_score × 100)   ← current state health
        + 50% × health_trend_score              ← long-term trend health
  Also derives: overall_status, maintenance_priority, issues list

STEP 10 — REACT FRONTEND (port 3000)
  Single-page app. All API calls proxy through port 4000.
  Components poll the backend on timers:
    - Latest sensor data:   every 2,500 ms (live readings)
    - Historical charts:    every 4,000 ms (history arrays)
    - ML analytics:         every 20,000 ms (model outputs)

 SECTION 4 — THE 9-FEATURE VECTOR (Shared by Both Models)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The key architectural decision: instead of each model watching one sensor,
BOTH models receive a 9-feature vector computed from ALL 4 sensors together.
This enables cross-sensor correlation — the most powerful diagnostic signals.

  ┌────┬──────────────────────┬──────────────────────────────────────────────┐
  │ #  │ Feature              │ What it captures                             │
  ├────┼──────────────────────┼──────────────────────────────────────────────┤
  │ F1 │ avg_rpm              │ Mean motor speed over window W               │
  │ F2 │ rpm_std              │ RPM stability — high std = erratic motor     │
  │ F3 │ avg_spm              │ Mean stitches per minute — production rate   │
  │ F4 │ spm_to_rpm_ratio     │ avg_spm / (avg_rpm + ε) — THE KEY FEATURE   │
  │    │                      │ RPM=1400, SPM=0 → ratio=0 → thread jam      │
  │    │                      │ catches faults no single sensor can detect  │
  │ F5 │ vibration_rms        │ sqrt(mean(vibration_stable²)) — RMS energy  │
  │ F6 │ vibration_std        │ Vibration consistency — erratic = fault     │
  │ F7 │ avg_temp             │ Mean temperature over window                │
  │ F8 │ temp_rise_rate       │ °C/min from polyfit slope — rising = danger │
  │ F9 │ humidity             │ Latest humidity % — affects motor insulation│
  └────┴──────────────────────┴──────────────────────────────────────────────┘

  File: analytics/analyzers/features.py
  Window size W = 15 readings ≈ 75 seconds of machine data per feature vector
  Called by BOTH anomaly.py (Model 1) and degradation.py (Model 2)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────────────────────────────────────────────
 MODEL 1: ISOLATION FOREST — Real-time Multi-sensor Anomaly Detection
 File: analytics/analyzers/anomaly.py
 Endpoint: GET /analytics/anomaly/{machine_id}
────────────────────────────────────────────────────────────────────────────────

PURPOSE:
  "Is the machine behaving abnormally RIGHT NOW?"
  Detects when the current machine state (across all 4 sensors together)
  is statistically unusual compared to its recent normal behavior.

ALGORITHM — HOW ISOLATION FOREST WORKS:
  Isolation Forest (Liu et al., 2008). Unsupervised anomaly detection.
  Core insight: anomalies are rare and different — they are EASIER TO ISOLATE.

  1. Fetch last 200 readings from each of 4 sensor types.
  2. Apply shared feature engineering (features.py):
       For each sliding window of W=15 readings (stride=1):
         Compute the 9-feature vector.
         Append to feature matrix X.
       X shape = (N_windows, 9) where N_windows ≈ 40-185 windows
  3. Train: IsolationForest(n_estimators=100, contamination=0.05).fit(X)
       Build 100 random decision trees on the feature matrix.
       At each tree node: pick a random feature, pick a random split value.
       Keep splitting until each sample is isolated in its own leaf node.
       Count path length to isolate each sample across all 100 trees.
       Short path = anomaly (easy to isolate = different from others).
       Long path  = normal  (hard to isolate = similar to most data).
  4. Score the LATEST window (X[-1]):
       raw_score  = model.score_samples(X[-1:])  ← lower = more anomalous
       anomaly_flag = model.predict(X[-1:]) == -1
       Normalize to 0-1: anomaly_score = 1 - (score - min) / (max - min + ε)
  5. Feature importance (explainability — deviation analysis):
       mean_i = mean(X[:, i]) across all N windows
       std_i  = std(X[:, i]) across all N windows
       z_i    = (X[-1, i] - mean_i) / (std_i + ε)    for each feature i
       |z_i| > 2 = that feature is contributing to the anomaly
       top_contributors = features sorted by |z_i| descending, top 3 shown
  6. Map features to sensor components:
       avg_rpm, rpm_std              → "rpm" component
       avg_spm, spm_to_rpm_ratio     → "stitches" component
       vibration_rms, vibration_std  → "vibration" component
       avg_temp, temp_rise_rate, humidity → "temperature" component
       component_flags = which components have features in top_contributors

COLD START (< 10 running windows):
  Falls back to z-score across all 9 features (using running windows only).
  max_abs_z = max |z_i| across all 9 features; norm_score = min(1.0, max_abs_z/4.0)
  Anomaly if max_abs_z > 2.5.

ANOMALY THRESHOLDS:
  Score 0.00 – 0.49 → "normal"
  Score 0.50 – 0.69 → "watch"
  Score 0.70 – 0.84 → "warning"
  Score 0.85 – 1.00 → "critical"

IDLE MACHINE DETECTION (sewing machine stop is NORMAL, not anomalous):
  A sewing machine regularly stops between operations (fabric repositioning,
  thread changes, operator breaks). Idle stops must NOT trigger anomaly alerts.

  IDLE_RPM_THRESH = 50.0 (avg_rpm feature, index 0 in feature vector)

  Logic flow:
    1. If current_rpm (X[-1, 0]) < 50 → immediately return idle_result
       idle_result: anomaly_flag=False, anomaly_score=0.0, status="idle",
                    top_contributors=[], model="idle_bypass"
    2. Filter feature matrix to running windows only:
       X_running = X[X[:, 0] >= 50.0]
    3. Train Isolation Forest ONLY on X_running
       → idle stops do not distort the baseline for running behavior

  Why this matters: Without idle detection, idle windows (avg_rpm≈0, avg_spm=0)
  look like extreme outliers against the running baseline, causing the model to
  score them as CRITICAL. A 50 RPM threshold cleanly separates idle from running.

TRAINING STRATEGY — ROLLING WINDOW RETRAINING:
  We retrain on the last 200 readings every API call. This is adaptive baseline.
  Why not train once and persist?
    - A machine after maintenance has a new normal — should not be anomalous.
    - Rolling window automatically adapts to the current machine state.
  Why not train on ALL historical data?
    - A machine from 3 months ago may have been poorly tuned.
    - Anomaly = unusual for THIS machine NOW, not its entire history.

PARAMETERS:
  n_estimators  = 100   (100 trees for robust, stable scoring)
  contamination = 0.05  (assume ≤5% of readings are genuinely anomalous)
  random_state  = 42    (reproducible results for reports)
  Window W      = 15    (15 readings × 5s = 75s per feature snapshot)

OUTPUTS:
  anomaly_flag      (bool) — True if current state is anomalous
  anomaly_score     (float 0-1) — higher = more anomalous
  status            ("normal" | "watch" | "warning" | "critical" | "idle")
  top_contributors  (list of {feature, z_score, component})
  component_flags   ({rpm, stitches, vibration, temperature} → bool)
  feature_vector    (the 9 raw feature values for the latest window)
  feature_z_scores  ({feature_name: z_score} for all 9 features)
  model             ("isolation_forest" | "zscore_fallback")
  samples_used      (int — number of windows in training matrix)


────────────────────────────────────────────────────────────────────────────────
 MODEL 2: LINEAR REGRESSION — Long-term Degradation Trend Detection
 File: analytics/analyzers/degradation.py
 Endpoint: GET /analytics/degradation/{machine_id}
────────────────────────────────────────────────────────────────────────────────

PURPOSE:
  "Is the machine slowly getting WORSE over time?"
  Detects gradual degradation that builds up over hours or days — wear that
  never triggers an anomaly at any single reading but IS shortening the
  machine's life. Invisible to Model 1. Only visible with trend analysis.

WHY LINEAR REGRESSION FOR DEGRADATION?
  Mechanical wear follows approximately linear progression (Archard's wear law).
  OLS regression gives a slope interpretable directly: "vibration increasing
  by 8.4 units per hour". The slope projects forward to failure: "3000 units
  threshold reached in 7.3 hours." Fully explainable — no black box.
  No training data needed. No labeled failure examples required.
  R² tells you how clean the trend is: high R² = real degradation, not noise.

THE 5 TREND INDICATORS (Model 2's input features, one per bucket):
  ┌────┬──────────────────────────┬────────────────────────────────────────────┐
  │ T1 │ vibration_level          │ mean(vibration_stable) per bucket          │
  │    │                          │ Rising over time → bearing wear, imbalance │
  ├────┼──────────────────────────┼────────────────────────────────────────────┤
  │ T2 │ temperature_level        │ mean(temperature_c) per bucket             │
  │    │                          │ Slowly rising baseline → cooling degrading │
  ├────┼──────────────────────────┼────────────────────────────────────────────┤
  │ T3 │ spm_efficiency           │ mean(spm) / (mean(rpm) + ε) per bucket     │
  │    │                          │ Falling ratio → mechanical wear in needle, │
  │    │                          │ increasing thread drag over sessions        │
  ├────┼──────────────────────────┼────────────────────────────────────────────┤
  │ T4 │ rpm_stability            │ 1 / (std(rpm) + 1) per bucket              │
  │    │                          │ INVERSE of RPM variability.                │
  │    │                          │ Declining stability → power degrading       │
  ├────┼──────────────────────────┼────────────────────────────────────────────┤
  │ T5 │ composite_anomaly_score  │ Model 1's IF score applied to each bucket  │
  │    │                          │ Rising scores over time = machine gets      │
  │    │                          │ progressively harder to classify as normal  │
  └────┴──────────────────────────┴────────────────────────────────────────────┘

  NOTE on T5 (composite_anomaly_score):
  This is the most powerful trend indicator. Model 1's feature engineering
  is applied to each historical bucket, then scored with the Isolation Forest.
  A rising sequence of anomaly scores over hours = clearest degradation signal
  even if no individual reading ever crossed a threshold.

HOW IT WORKS STEP BY STEP:
  1. Fetch last 500 readings per sensor from MongoDB.
  2. Sort all by received_at. Divide into N_BUCKETS = 10 equal time segments.
  3. For each bucket:
       Compute: vib_level, temp_level, spm_eff, rpm_stability
       Mark bucket as idle if avg_rpm < 50.0 (is_idle = True)
       Apply feature engineering to bucket data → score with Isolation Forest
       → anomaly_score for this historical period
       Append [vib_level, temp_level, spm_eff, rpm_stability, anomaly_score]
       Result: trend_matrix shape (10, 5)

  IDLE BUCKET EXCLUSION:
       For spm_efficiency and rpm_stability regression:
         Only use non-idle buckets (avg_rpm >= 50). Idle periods have
         spm=0 and rpm=0 which would pull the regression slope negative,
         making a healthy machine appear to be "degrading" in efficiency.
         Minimum 3 non-idle buckets required; otherwise slope=0.0 (stable).
       vibration_level, temperature_level, anomaly_score: use all buckets
         (these are valid to measure even during idle periods).

  4. For each of the 5 columns in trend_matrix:
       slope, intercept = polyfit(range(10), trend_matrix[:, j], 1)
       r_squared = pearsonr(range(10), trend_matrix[:, j])[0]²
  5. Normalize each slope to a degradation contribution:
       vibration:    contribution = slope / VIBRATION_DANGER (3000 units)
       temperature:  contribution = slope / TEMP_DANGER (60°C)
       spm_eff:      contribution = -slope  (negative slope = degrading)
       rpm_stability: contribution = -slope  (negative slope = degrading)
       anomaly_score: contribution = slope   (already 0-1 scale)
  6. degradation_rate = mean(all 5 contributions) × 100  (%/hour scale)
  7. health_trend_score = max(0, min(100, 100 - degradation_rate × 10))
  8. projected_failure_hours = minimum across all indicators:
       For each indicator: readings_to_danger = (danger - current) / slope
       Convert to hours using bucket duration.
  9. early_warning = True if ANY of:
       degradation_rate > 2.0 (rapid worsening)
       health_trend_score < 70 AND anomaly_score_trend slope > 0
       vibration trend projects failure within 7 days
       temperature trend projects crossing 60°C within 24 hours
       R² of any critical indicator > 0.8 (clean trend = real degradation)

HEALTH TREND SCORE INTERPRETATION:
  100     → No degradation detected, all trends stable or improving
  80-99   → Minor degradation — monitor
  60-79   → Moderate degradation — plan maintenance soon
  40-59   → Significant degradation — schedule maintenance this week
  0-39    → Rapid degradation — urgent maintenance required

OUTPUTS:
  health_trend_score      (0-100, 100 = no degradation)
  degradation_rate        (%/hr, positive = worsening)
  trend_status            ("stable" | "mild" | "moderate" | "significant" | "rapid")
  early_warning           (bool)
  projected_failure_hours (float or None)
  per_indicator_trends    ({indicator: {slope, r_squared, direction, projected_hours}})
  bucket_data             (list of 10 bucket summaries — used for sparkline charts)
  buckets_analysed        (int)
  samples_used            (int)


────────────────────────────────────────────────────────────────────────────────
 COMBINED: SYSTEM HEALTH (system.py)
 Endpoint: GET /analytics/system/{machine_id}
────────────────────────────────────────────────────────────────────────────────

PURPOSE:
  Single endpoint that runs BOTH models and combines their outputs into
  one unified machine health assessment. This is what the Analytics.jsx
  dashboard page calls.

MACHINE HEALTH SCORE (MHS) FORMULA:
  MHS = 50% × (100 - anomaly_score × 100)   ← current state (Model 1)
      + 50% × health_trend_score              ← long-term health (Model 2)
  Range: 0-100

  MHS interpretation:
    80-100 → HEALTHY  (all systems operating normally)
    65-79  → WATCH    (minor issues, monitor closely)
    45-64  → WARNING  (significant issues, plan maintenance)
    0-44   → CRITICAL (stop machine, immediate inspection)

IDLE STATUS PROPAGATION:
  If Model 1 returns status="idle" (machine currently stopped):
    → overall_status = "idle" regardless of MHS value
    → Machine is never classified as "critical" while stopped
  This prevents false alarms when operators pause the sewing machine between
  operations, which is a completely normal part of the sewing workflow.

MAINTENANCE PRIORITY:
  MHS < 30 OR rul_hours < 1  → URGENT
  MHS < 50 OR rul_hours < 4  → HIGH
  MHS < 65 OR 2+ issues      → MEDIUM
  MHS < 80                   → LOW
  MHS >= 80                  → NONE

ISSUES LIST:
  Aggregates anomaly contributors from Model 1 (which features deviated)
  and degradation indicators from Model 2 (which indicators are trending bad).
  Ranked by severity (critical first, warning, watch, info).

OUTPUTS:
  machine_health_score  (0-100)
  overall_status        ("healthy" | "watch" | "warning" | "critical" | "offline" | "idle")
  maintenance_priority  ("none" | "low" | "medium" | "high" | "urgent")
  rul_hours             (projected_failure_hours from Model 2, or None)
  issues                (list of {component, severity, message} ranked by severity)
  anomaly               (full Model 1 output)
  degradation           (full Model 2 output)

 SECTION 9 — KEY DESIGN DECISIONS EXPLAINED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: Why 2 models instead of the original 4?
A: The original 4 models each watched one sensor in isolation. They could not
   detect cross-sensor anomalies (e.g., RPM=1400 but SPM=0 → thread jam —
   missed by RPM model, missed by temperature model, only caught by stitches
   model, but no single model saw the full picture). The 2-model design sees
   ALL 4 sensors together via the 9-feature vector, enabling cross-sensor
   correlation. It also adds long-term degradation detection (Model 2) which
   the 4-model system completely lacked.

Q: Why is ML in Python and not in Node.js?
A: Scikit-learn, NumPy, and SciPy are mature Python libraries with no
   equivalent in Node.js. Python has dominated scientific computing for
   20 years. Node.js is better at real-time I/O (MQTT, API).
   Each language is used where it is strongest.

Q: Why does the Node backend proxy to Python instead of the frontend calling
   Python directly?
A: Single origin for the frontend. The React app only needs to talk to
   localhost:4000. If Python moves to a different server, only the proxy
   URL changes — zero frontend changes needed.

Q: Why poll every 20 seconds for ML and 2.5 seconds for live data?
A: Feature engineering + IsolationForest training on 200×9 matrix takes
   ~50-200ms. Doing it every 2.5 seconds would unnecessarily load CPU and
   MongoDB. Live sensor values are in-memory reads — 2.5s is fine.

Q: Why retrain the Isolation Forest on every API call?
A: The model trains on the LAST N readings, so it adapts to the current
   machine state automatically. After maintenance the machine has a new
   normal — a model trained yesterday would incorrectly flag normal readings.
   Rolling window retraining is the correct approach for streaming sensor data.

Q: Why contamination=0.05 in the Isolation Forest?
A: Tells the model to assume ≤5% of readings may be anomalous.
   Too low (0.01) → misses real anomalies.
   Too high (0.20) → too many false alarms.
   5% is the standard industry default for mechanical sensor data.

Q: Why Linear Regression for degradation and not a more complex model?
A: Mechanical wear IS approximately linear over time (Archard's law).
   The slope maps directly to a rate ("8.4 vibration units per hour").
   Projecting to failure is simple division. R² gives confidence level.
   No training data required. Fully explainable in a viva or report.
   A more complex model would give the same answer with 100× more complexity
   and 0 additional interpretability.

Q: Why use composite_anomaly_score (T5) as a trend indicator?
A: Running Model 1 on each historical time bucket and plotting the resulting
   anomaly scores over time reveals if the machine is progressively harder
   to classify as normal. A rising sequence of anomaly scores over hours
   is the clearest possible signal of gradual degradation — even if no
   individual reading ever crossed the anomaly threshold.

Q: What happened to the old EWMA, stitches forecast, etc.?
A: Those algorithms are still in the codebase (rpm.py, stitches.py, etc.)
   but are no longer called by main.py v2.0. Their analytical value is
   preserved — EWMA signals are captured by avg_rpm/rpm_std in the feature
   vector; stitch forecasting insights appear in spm_to_rpm_ratio and
   spm_efficiency trend; vibration Isolation Forest is now applied to all
   9 features not just 4 vibration features; temperature rate-of-change is
   captured by temp_rise_rate in the feature vector.

 SECTION 12 — VIBRATION THRESHOLDS (Consistent Across All Components)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The MPU6050 accelerometer stores vibration_stable as raw ADC units.
The ±2g scale gives: 1 g-force = 16384 ADC units.

THRESHOLD STANDARD (applied consistently across ALL frontend pages):
  Normal   : vibration_g < 0.10 g    (raw < 1638)
  Warning  : vibration_g 0.10–0.20 g (raw 1638–3277)
  Critical : vibration_g ≥ 0.20 g    (raw ≥ 3277)

CONVERSION:  vibration_g = vibration_stable / 16384

These thresholds apply in:
  • server.js              — real-time MQTT alert generation
  • VibrationMonitoring.jsx — KPI card, chart reference lines, status label
  • Overview.jsx           — vibration KPI card, priority alert trigger
  • MaintenanceAlerts.jsx  — KPI card, vibration chart reference lines
  • backend alerts API     — /api/alerts/:machineId threshold checks

IMPORTANT: server.js alert logic converts to g-force BEFORE comparing:
  const vibG = latest.vibration_stable / 16384;
  if (vibG >= 0.2) → critical alert
  if (vibG >= 0.1) → warning alert

Chart display: all vibration charts show values in g-force (after /16384),
capped at G_MAX_CHART = 0.3 g on the VibrationMonitoring page.

  Fix    : Added conversion before comparison:
